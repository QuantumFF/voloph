# Transcode web-incompatible recordings in place

Recordings the webview cannot decode (notably iPhone HEVC, which WebKitGTK refuses) are made playable by transcoding them **once, in place, to H.264/AAC**: the transcoded file replaces the original at its existing path and the source codec is discarded. Recordings already in a web-playable codec (H.264/AAC in an mp4/mov container) are left untouched. There is never a second copy of a recording.

Chosen over keeping a separate cached proxy alongside the untouched original (the approach this supersedes) because a proxy means storing two near-identical copies of a multi-gigabyte recording — the original the webview can't use and the transcode it can. For footage imported specifically to be reviewed in this app, the original codec carries no further value once a seekable H.264/AAC version exists, so paying double the disk to retain it is waste, not safety. A single in-place file keeps the reference-in-place model (the recording stays at its path, merely re-encoded) and gives the seek-dominated review loop a complete, frame-accurate, known-duration file.

The transcoded file is served to the webview `<video>` element over a small **loopback HTTP server** (`127.0.0.1`, ephemeral port, per-launch token) that declares `Content-Type: video/mp4` and honors HTTP range requests — **not** the asset protocol or a custom URI scheme. WebKitGTK plays HTML5 media through GStreamer, which loads real `http://` sources (with range seeking via `souphttpsrc`) but does not route media through WebKit's custom-scheme handlers, so a `<video>` pointed at `asset://`/`stream://` fails with `MediaError` code 4 (`SRC_NOT_SUPPORTED`), `networkState` 3, even when the bytes are valid H.264/AAC. Serving over loopback HTTP with an explicit `video/mp4` type sidesteps both the codec problem and the extension-derived-MIME problem, without renaming files (no path churn) and without a second copy.

The transcode forces a keyframe roughly once per second (`-force_key_frames "expr:gte(t,n_forced*1)"`), overriding x264's sparse default GOP (~250 frames ≈ 8s at 30fps). Measuring a transcoded clip confirmed the default left keyframes ~8s apart, so any scrub to a point between them had to decode forward from a distant keyframe — the "buffering / not responding" feel while scrubbing. Forcing ~1s keyframes lets an arbitrary seek land near a keyframe, giving frame-accurate, smooth scrubbing in the seek-dominated review loop; the expression is fps-independent and preserves source frame rate, and the modest file-size increase is accepted as the right trade for the review loop. (The transcode is already a lossy re-encode, so this does not conflict with the "never re-encode already-web-playable H.264" rule — H.264 source still passes through untouched, including the minor iPhone "Most Compatible" case, which therefore does not gain the dense keyframes.)

The transcode runs in the background at import, alongside audio segmentation (ADR 0002), so a recording is normally ready before the user opens it; until it lands the UI marks the recording as still processing rather than handing the player an undecodable original. Each transcode writes to a temporary file in the same directory and atomically renames over the original, so an interrupted run never leaves a recording half-written. After it lands, the recording's stored file size and quick hash (ADR 0003) are refreshed to match the new bytes.

Constraints to respect, not visible in code: this is **destructive** — the original codec is gone afterward. It is the one deliberate exception to "original files are never modified" (ADR 0001), justified narrowly as one-time import normalization, never as editing; review itself still never alters a file. It costs CPU at import and re-encodes lossily, so only web-incompatible recordings are transcoded — H.264/AAC recordings are never re-encoded, avoiding needless quality loss. ffmpeg/ffprobe are the bundled sidecars (ADR 0004).

Supersedes two earlier playback approaches, both now removed: the live transcode-and-stream loopback server (issue #15, commits `63ecc50`, `01e46eb`) and the cached-proxy-alongside-original design that briefly replaced it (issue #16). ADR 0004's decision to bundle ffmpeg as a sidecar still stands.
