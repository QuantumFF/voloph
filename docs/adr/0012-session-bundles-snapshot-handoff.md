# Share review state as session bundles: metadata snapshots handed off through the library root

Review state travels between devices and people as a **session bundle** — a small metadata-only file snapshotting one session's review: per recording, its library-relative path, quick hash + size, capture day, duration, waveform, the full timeline (including machine confidence and uncertain regions), annotations, and flags. No video is ever carried; recordings resolve against the recipient's own mount of the same shared library (ADR 0011). Bundles exist only in the **shared** library — sharing from the local library is meaningless, as recipients cannot reach its files. **Share** writes the bundle into the shared library itself, keyed by session + sharer label — re-sharing overwrites your own previous bundle, so there is one current bundle per person per session. Recipients discover bundles automatically when scanning the shared library and are offered them by name; saving a bundle elsewhere and opening one manually remain as fallbacks.

Receive is **self-sufficient**: it can register recordings the recipient never scanned — verify the file exists, apply the carried state — so a recipient hits play seconds after mounting, with no probe and no segmentation of hours of video over a network mount. Verification is strict: quick hash + size match → accept; mismatch → refuse that recording (the rest of the bundle still applies) with a message naming the file. There is no innocent mismatch to absorb — the in-place transcode that once rewrote bytes (ADR 0005) was eliminated by ADR 0008, so a mismatch means the file at that path is not the recording the bundle describes. Aspects unknown to the recipient are silently added to their vocabulary (it is user-editable by design; stripping them would mutilate the shared review).

A bundle is a **snapshot handed over, not a collaboration channel**: received state becomes the recipient's own, with no attribution or provenance (a note can carry a signature). Where the recipient has already hand-touched a recording's review, they choose keep-mine-or-take-theirs per recording; nothing merges — annotations could merge additively, but a recording has exactly one timeline, and half-merging two people's boundary corrections is worse than either original. Receiving the same bundle twice is a no-op.

Chosen over: **copying `voloph.db` to the root** — whole-library last-writer-wins silently erases whichever device pushed first and drags per-device state along; **live sync / per-recording auto-sync files** — a conflict-resolution problem this single-reviewer-per-session workflow doesn't need; **bundles carrying video** — gigabyte archives, chunking, and a managed import library (re-opening what ADR 0003 rejected), when a recipient without storage access can simply be given access to the root; **attribution / multi-reviewer layers** — a real feature deserving its own design if ever needed, not a speculative `author` column.

This does **not** reverse ADR 0003's "SQLite, not JSON sidecars": SQLite remains the sole working store and source of truth on every device; bundles are a transfer format written to and read from the root, and cross-recording relational queries still run against the database. The bundle format itself is the hard-to-reverse part — bundles live in users' shared folders, so format changes must remain readable backward.
