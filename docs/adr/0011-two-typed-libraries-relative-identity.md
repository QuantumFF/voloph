# Two typed libraries (shared + local); recordings identified relative to their library; mount is per-device

All recordings live under a **library** — a user-designated folder that is the app's whole world while active; a video outside it does not exist to the app. Ad-hoc folder scanning (the `scanned_folders` model) is replaced by scanning the active library. There are **at most two libraries, one of each type**: a **shared library** on common storage (NAS, or Google Drive via any mounting tool) that other people and devices reach, and a **local library** private to the device — the home of the "quick same-day review of one file" journey without touching the NAS. A switcher makes one active at a time; the session list, filters, and review see only the active library, and a session never spans libraries — which is what keeps the portable/non-portable two-class problem structurally impossible. Reference-in-place (ADR 0003) is unchanged — the user puts files in a library; the app never copies, moves, or rewrites them.

A recording's identity is its **path relative to its library's folder**; the absolute path is computed at use time. For the shared library, each device declares where it is mounted *here* and whether that mount is **local or network** (an explicit user choice, not filesystem detection — FUSE/rclone mounts masquerade as local, and the user may want staging off on fast wired NAS). Locality is a per-device fact, not a property of the library: the same shared folder is a local disk on the NAS-host machine and a network mount on a laptop. Nothing about mount location or locality ever enters shared metadata. The app stays purely filesystem-based — "cloud support" is a mounted folder, not a Drive API integration, which would force OAuth and download/cache layers in front of libmpv/ffmpeg for multi-gigabyte files those subsystems want as real local paths.

Chosen over the alternatives: a **single root** (the design's first draft) killed the quick-local-review journey — every file had to enter the central store before the app could see it; **arbitrarily many roots** forces a cross-device root-identity scheme (how does device B know root #2 is device A's root #2?) and buys nothing over one-of-each-type; **keeping absolute paths and re-matching by quick hash** makes every other device's stored path dead weight and re-reads 64 KB per file per scan over the network; **content-addressed identity (hash only)** still needs a per-device hash→path index — it reduces to relative paths with extra steps. The quick hash (ADR 0003) is retained for verification and re-locating moved files.

Designating (or re-designating) a library runs an **adoption pass**: every already-known recording found under it — by dead-path prefix, or by quick hash + size for moved files — is converted to that library's relative identity with all review state intact; known recordings not found are reported unresolved rather than silently kept as a second class. The same machinery works **across libraries**: a recording that vanished from one library and matches by hash + size in the other is re-homed automatically (a move), and when the same content exists in both (a copy) with hand-touched review state on one side only, the app **offers** — never silently — to carry the review over. This completes the intended journey: quick-review locally, copy the session to the NAS, bring the review along, share the bundle.

On a shared library whose mount is declared **network**, analysis (probe, audio pass, motion pass, waveform) does not stream the file across the mount repeatedly — the pipeline makes at least two full reads per recording (`media.rs`: separate ffmpeg runs for 16 kHz audio and 5 fps frames). Instead the file is **staged**: copied once into a bounded local staging area, analyzed locally, results written to SQLite, staged copy evicted — batched (copy-next-while-analyzing-current) so a 100 GB session needs only the staging budget, not 100 GB. Staging is skipped entirely when a session bundle (ADR 0012) already carries the analysis, and re-engaged by an explicit re-analyze.

Consequences: portability is a property of which library a recording lives in, not app-enforced — the local library is a perfectly valid world no other device can see, and session bundles (ADR 0012) exist only in the shared library. Playback quality over a slow mount (e.g. an uncached rclone Drive mount) is a documentation/support concern, deliberately not an app feature.
