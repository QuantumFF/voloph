# Seek by reloading a server-side byte-offset stream

> **Status: superseded by ADR 0008.** This whole mechanism existed only because
> WebKitGTK silently drops `currentTime` seeks. Playback moved to embedded
> libmpv, which seeks natively, so the seek-by-reload stream, the double buffer,
> the JPEG frame-step, and the loopback server are all removed. The diagnosis
> below remains the record of *why* the webview path was abandoned.

Seeking the player to a position is done by **reloading the `<video>` at a `&t=<seconds>` URL** that the playback server answers with a fragmented MP4 beginning at the keyframe at or before `t` — **not** by writing `media.currentTime`. Recording-local time is then `seekBaseMs + media.currentTime`, where `seekBaseMs` is the `t` the stream was opened at; the frontend threads that base through every place playhead time is read or written (the timeline playhead, gap-skip, rally bounds, the session global/local conversion). The whole-file path (`t == 0`, range-seekable) is kept for opening a recording at its head; every non-zero seek — a timeline click, an arrow-key jump, gap-skip into the next rally, a rally-loop restart, a boundary crossing — reloads a fresh `&t=` stream.

Chosen because **`currentTime` seeks are silently dropped on this WebKitGTK/GStreamer build** (2.52.4 / 1.28.4): the write is ignored and `seeked` fires back at the pre-seek position, even on a fully-buffered local file seeking a few seconds forward. This was reproduced interactively across ~8 rounds and every plausible frontend cause was ruled out with a live repro — HTTP range serving, buffering/`seekable` range, keyframe density (a dense 1s-keyframe transcode failed identically), moov position, `souphttpsrc` streaming mode, our pause/play machinery, server concurrency, missing plugins (issue #24, escalated from #23/#20). No frontend retry converges, so the player must avoid issuing a GStreamer seek at all: a stream that *opens* already positioned never seeks.

The server produces the positioned stream with `ffmpeg -ss <t> -i <path> -c copy -movflags frag_keyframe+empty_moov -f mp4 pipe:1` (the bundled sidecar, ADR 0004). `-ss` before `-i` is a fast keyframe seek and `-c copy` remuxes without re-encoding, so the cut is cheap and uses almost no CPU — which matters because the review loop reloads a fresh stream on every seek; `frag_keyframe+empty_moov` yields a streamable fragmented MP4 whose moov is up front, so it plays progressively over a pipe — `+faststart` is deliberately **not** used, as it needs a seekable output and cannot stream. ffmpeg's stdout streams straight to the response body (chunked, length unknown ahead of time); the child is killed and reaped when the webview drops the connection, which a superseding seek does. Output timestamps reset to ~0, so recording-local time is `seekBaseMs + media.currentTime` with `seekBaseMs = t`.

Because a `-c copy` cut cannot start mid-GOP, the stream begins at the keyframe ≤ `t`, so the first frame can be up to one keyframe-interval before the labelled position. This is only acceptable while **every** recording is kept at ~1s keyframes at rest, so the snap is ≤~1s (the tolerance scrubbing already lived with). The forced ~1s keyframes of ADR 0005 originally applied only to codec-incompatible files; a web-playable recording was passed through untouched and kept its camera's native GOP, commonly many seconds (the real recordings here run ~8s). On those, every short arrow-key seek that landed inside one GOP snapped to the same keyframe and replayed the identical scene — seeking appeared to do nothing (issue #24, found after the initial implementation). The fix broadens the import transcode (ADR 0005) to also re-encode web-playable-but-sparse recordings to dense keyframes, gated by a keyframe-gap probe, so copy-based seeking is accurate everywhere; a one-time `user_version` migration re-probes recordings imported before the gating. Re-encoding the stream *per seek* to land exactly at `t` was tried and rejected: it is frame-accurate, but re-encodes 1080p on every reload and must decode a whole GOP to reach `t`, so scrubbing was laggy and CPU-heavy — densifying once at import keeps every seek a cheap copy.

Constraints to respect, not visible in code: every seek is a **fresh stream load** — a visible reload plus ffmpeg-startup latency — including the frequent gap-free and rally-loop seeks, where a reload flashes between iterations. This is accepted for a local, single-user review tool as the cost of reliable seeking on this platform; it is the simplest correct shape (route all seeks through `&t=`). A hybrid that keeps `currentTime` for seeks that happen to land was left for later only if the reload proves too costly. **Frame-step gets its own path, not `currentTime`**: it was first carved out as the one exception — a direct `currentTime` nudge, on the bet that a paused one-frame move was "least likely to hit the drop" — but that bet was wrong (the drop is unconditional on this build), so frame-step looked frozen while the playhead moved. It now never touches the `<video>`: the server decodes the exact frame (`/play?...&frame=1` → `ffmpeg -ss <t> -i <path> -frames:v 1 -f mjpeg`, an accurate decode forward from the keyframe, so it is frame-accurate where the `-c copy` stream can only start on a keyframe) and the player overlays that JPEG over the paused video. The still is preloaded and the visible `<img>` swapped only once decoded (no black gap), requests are coalesced so holding `,`/`.` advances at decode speed instead of starving, and resuming play reopens the `&t=` stream at the stepped frame, so the overlay is only ever shown while paused (issue #19). `media.duration` of a `&t=` stream is the *remaining* duration, not the recording's — the player never relies on it (duration comes from the segmented timeline in the DB), so this is fine.

Supersedes the assumption in ADR 0005 that loopback-HTTP range serving gives "native frame-accurate seeking" — it does for byte ranges, but the GStreamer seek built on them is unreliable on this build, which is why seeking now reloads rather than seeks. The loopback server, token, and `video/mp4` transport (ADR 0005) and the ffmpeg sidecar (ADR 0004) are unchanged.
