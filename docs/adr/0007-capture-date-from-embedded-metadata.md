# Derive a session's capture date from embedded metadata, not file mtime

Recordings are grouped into **sessions** by capture date (ADR 0003: same calendar day = one session). That date is now read from the **creation date the camera embedded in the recording**, falling back to the file's modified time (mtime) only when no usable embedded date exists. Grouping is refined in the background after a recording is registered, and re-homing a recording to a different day moves it to the matching session (creating it, and garbage-collecting an emptied one) in one step.

Chosen over deriving the day from mtime — the previous stopgap — because mtime is content-independent and routinely wrong: copying off a camera or SD card, cloud sync, and the app's own in-place transcode (ADR 0005) all stamp mtime to the copy/transcode time, scattering a single outing's recordings across the wrong days. The embedded creation date is written once by the capture device and rides with the bytes, so it survives every operation that clobbers mtime — provided the transcode preserves it, which is why ADR 0005 now maps metadata across the rewrite.

The day is taken from the first source that yields a *plausible* date, in order: `com.apple.quicktime.creationdate` (iPhone footage — local wall-clock **with** the camera's UTC offset), then `creation_time` (ISO-8601 in UTC), then mtime. The offset-bearing tag is preferred because a session is the player's local outing: its date portion is already the local day, so a late-evening session stays on its own day instead of rolling into the next one as a UTC timestamp would. A value that is absent, unparseable, or implausible (the `0000-00-00` a metadata-stripped transcode used to leave behind, a dead-clock year) is skipped rather than trusted, so a bad tag falls through to the next source instead of mis-grouping.

Grouping runs as a background work phase (ahead of probe/transcode/segment) rather than synchronously at scan, so a folder scan stays responsive: the recording is registered immediately under a provisional mtime-derived day and re-homed once its embedded date has been read. The phase is gated by a per-recording `date_state` flag that defaults to "unknown", so adding it to an existing database makes every already-imported recording eligible for re-derivation on the next pass — a **backfill** that re-groups recordings imported under the old mtime rule, with no separate migration step. Rallies and annotations key off the recording, not the session, so they travel with it when it moves.

Accepted weakness: a recording transcoded *before* the metadata-preserving fix (issue #25) had its embedded date wiped **and** its mtime clobbered to the transcode time, so neither source is correct — it falls back to that wrong mtime and nothing retroactive recovers it. Going forward the transcode preserves the date, so this is a one-time cohort, not an ongoing loss. Second weakness: the UTC-`creation_time` fallback groups by UTC day, which can place a recording on the wrong local day near midnight when the offset-bearing tag is missing; this is strictly better than mtime and only affects cameras that omit the local-offset tag.
